# Final Review — Sancho Phase-0 Operator Kit

## Verdict

**READY-WITH-FIXES.** All 15 artifacts are present and non-stub; the closing documents (#12–14), PI-1, and the COGS arithmetic verify clean. But the kit fails several of the plan's own checkable acceptance clauses (#4 "structurally blocks", #10 "recomputes", the promised `.xlsx` sheets) and cannot pass its own fail-closed #15 gate as formatted. Every fix is mechanical and bounded (~1–2 builder-days); nothing requires redesign or new content.

## Synthesis

### Readiness Verdict

**READY-WITH-FIXES.** The plan's acceptance wording for #4 — "sheet structurally blocks A without amount >=10000 + 'non-refundable', and B without an attached document; tally formulas count only qualifying rows" — and its file rule "sheets `.xlsx`, snake_case columns" are not met as built: the ledger, target list, timesheet, and cost model are markdown tables with prose rules, and the runbook's own folder map names four `.xlsx` files that do not exist. These are real acceptance failures, but the content behind them (validation rules, formulas, canonical blocks) is fully written down, so the grade is READY-WITH-FIXES — usable next week after a short, mechanical fix pass — not NOT-READY (nothing is missing or gestured-at) and not READY (the kit's own final gate would return BLOCKED today).

### Blocking Issues

1. **p0-a-04 — the ledger cannot capture two of the numbers the day-90 verdict runs on (PASS Route 3 and a KILL limb).** The 13-column schema has no LOI field; the ledger punts: "LOI presence is recorded via status/evidence notes or a dedicated LOI flag outside the A/B/C tier when applicable" — yet dashboard #6 counter 3 and memo #7 §2/§3 require *ledger row IDs* for LOI counts, and none of the 8 fixtures exercises the LOI path. Separately, the tally defines presence as "`rfp_questionnaire_presence` | COUNT of rows with instrument = `B` (proxy: written insertion commitment implies questionnaire/RFP presence)", while #6 counter 5 defines it as "RFP/questionnaire presence (Y/N at program level)" and #5's quick reference as "RFP/questionnaire presence Y if doc exists." A buyer with a live questionnaire but no written commitment is Y on the dashboard and 0 in the ledger; since KILL requires "zero RFP/questionnaire presence," the divergence can falsely arm the KILL limb, and the memo cannot cite row IDs for presence because #4 has no presence column at all. **Fix:** add per-org `rfp_questionnaire_presence` (Y/N + document name, fed by screening Q4 / #5 Step 1) and `loi_signed` + `loi_evidence_link` columns (signed #14 = only qualifying evidence); add an `LOI_count` tally and one LOI fixture; delete the B-count proxy.

2. **p0-a-04 / p0-a-01 / p0-b-10 — "structurally blocks" is prose, and the promised workbooks don't exist.** The delivered ledger says "Structural blocks (sheet / scorer must enforce)" — scorer discipline, not structure; an invalid A row can simply be typed in. The runbook map names `p0-a-01-target-list.xlsx`, `p0-a-04-instrument-ledger.xlsx`, `p0-b-10-timesheet.xlsx`, `p0-b-10-cost-model.xlsx`; none is delivered, failing the plan's "sheets `.xlsx`" convention and #10's "cost model recomputes when any input changes." **Fix:** ship the workbooks with data validation (reject `instrument=A` unless `amount>=10000 AND refundability="non-refundable" AND evidence_link<>"none" AND signer<>"n/a"`; reject B without `evidence_link`) and COUNTIFS-style tallies over qualifying rows only; at minimum, append the exact validation/formula text so the founder can build them in under an hour.

3. **p0-a-05 §5.1/§5.2 and p0-a-03 — hard-rule paraphrase in buyer/vendor-facing copy; fails fail-closed #15 item 4.** Canonical clause 1 is `the sealed evaluation "cert bank" is never sold / trained on / customized per vendor`. The buyer readout (template and fixture fill) renders: "We do not sell or customize a sealed evaluation "cert bank" per vendor" — **"trained on" is dropped**, a substantive weakening of the integrity promise, and clauses 3–4 are reworded first-person ("we provide no cert-prep consulting; we never market detection beyond ~7% recall"). Vendor Touch 2 paraphrases ("We do not sell the sealed evaluation "cert bank," train on it, or customize it per vendor"), and the class-2 call open drops the quotation marks around "cert bank." The plan requires the five clauses "verbatim wherever touched," and #15 item 4 requires character-for-character matches with any No blocking release. **Fix:** either paste the canonical block verbatim in these spots, or state the constraints in plain prose without presenting them as the hard-rule clauses and record in #15 that first-person restatements are not hard-rule occurrences — one policy, applied consistently; in all cases restore "trained on" to the buyer readout.

The sentence bodies themselves are clean (ASCII `>=` throughout; thresholds semantically exact everywhere). **Fix:** strip emphasis and standardize one canonical rendering, *or* add an explicit normalization rule (strip markdown emphasis, collapse line breaks) to #15's character-diff procedure.

5. **p0-b-15 — the final release gate has never been run.** The pre-handoff log covers only #12–14; the "Final assembled-kit signed log" fields (kit hash, Yes-count, disposition) are blank, and the release rule says "Blank is `No`. ### Should-Fix

1. **p0-b-09 §1 vs §2.1 — `false-completion rate` denominator mismatch (confirmed by recompute; must land before the Wk 3–5 dry run and any paid run).** §1 defines C = Σ`completion_claim` (all claims); §2.1's formula is F/C×100, but its denominator prose is "completion claims with `acceptance_result` equal to `pass` or `fail`" and the missing-data rule excludes `not_adjudicable` from both sides. Recompute: A=10, C=8 with one claim `not_adjudicable`, F=2 → F/C = 25.0% but F/C_adjudicable = 2/7 ≈ 28.6%. The formula as symbolized is wrong whenever any claim is `not_adjudicable`; both worked examples recompute cleanly only because they contain zero such claims. **Fix (one line):** define `C_adj = COUNT(completion_claim=1 AND acceptance_result IN {pass, fail})`, use F/C_adj, and propagate through #10/#11 reporting. Not blocking for next week's outreach — Phase-0 PASS/KILL counts pilots/commitments/LOIs, not MP-1 values, and the prose disambiguates intent — but it corrupts the pilot deliverable if unfixed.
2. **p0-b-09 — acceptance miss: no synthetic raw task-level run log.** Acceptance requires an engineer to compute all five metrics "from a supplied synthetic raw run log"; both examples supply only pre-aggregated counts. Add a ~10-row log in the §1 schema that reproduces Worked Example A.

### Highest-Leverage Suggestions

1. **Fix the top-of-funnel math, or the test dies of starvation rather than falsified demand (plan omission).** The target sheet caps at 30 prospects while the runbook's Wk 2 calls for "~40–60 new touches" and 15 *completed* conversations with CISOs, VP Engs, and underwriters. Expand #1 to 100–150 capacity with per-class replenishment and add a funnel row (touches→replies→booked→completed, by class) to #6 so "no demand" and "no reach" separate by day 30.
2. **Ship one enforced evidence workbook (bundles Blocking 1–2).** A single `.xlsx` with the LOI columns, per-org RFP-presence column, A/B validation rules, and formula tallies is the artifact every PASS/KILL number rests on. Build it and re-run the 8 fixtures (plus one LOI fixture) against it before the first call.
3. **Name the second scorer and run Week-1 calibration (plan omission).** #1 (10 fixtures), #4 (8 fixtures), and memo #7's independent reviewer all require a second human that nothing assigns for a solo founder. Add to #8 Wk 1: recruit one named external scorer, run both fixture sets, log the agreement matrix as evidence.
4. **Give the day-30/60 checkpoints decision content (plan omission).** KILL is only evaluable at day 90 and PASS rarely closes early, so the checkpoints are inert. Add tripwires: day 30, <5 completed conversations or zero A/B asks delivered → forced one-named-pivot review; day 60, zero instrument-A term sheets in negotiation → pre-draft the KILL memo.
